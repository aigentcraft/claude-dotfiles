## 1. Plan / Context

記事・LP・Xスレッドで「読者に行動してほしい」のに動かない問題。
直接的な命令（「〜してみましょう」「あなたも〜できます！」「ぜひ試してみてください」）を使うと、**心理的リアクタンス**が発動して逆に行動が抑制される。

## 2. Do / The Error

- タイトルが「痛み」のみ → 共感は取れるが「で、どうすれば？」で離脱
- 結論を押しつける構成 → 「わかってる」「説教っぽい」と感じてスクロールアウト
- 「〜しましょう！」で締める → 読者の自律性を脅かすと認知され、反発が起きる
- Pain しか描かない → ベネフィット（得たい未来）が見えないと動き出す理由がない

## 3. Check / Root Cause

### 心理的リアクタンス（Psychological Reactance）
自由を制限されたと感じると、その行動への抵抗が増す（Brehm, 1966）。
「〜しましょう」「〜すべき」は、読者の自律性を脅かすと認知される。

### 自己決定理論（Self-Determination Theory）
人は「外から強制された行動」より「自分で気づいた行動」を取りやすい（Deci & Ryan, 1985）。
コピーの役割は「説得」ではなく**「気づきの場を設計すること」**。

### Pain × Benefit のバランス不足
- 痛みだけ → 「あるある」で終わる。行動エネルギーが生まれない
- ベネフィットだけ → 信頼されない（根拠・共感がない）
- 最適：**痛みで共感 → ベネフィット（得たい未来）で希望 → Curiosity Gap で引き込む**

## 4. Act / Prevention Strategy

### Future Pacing（未来ペーシング）
読者が「既に解決後の状態にいる自分」を想像できるシーンを描く。

❌ 「このツールを使えば速くリリースできます」
✅ 「次にアイデアが湧いたとき、電車を降りる前に push できている自分を想像してみてください」

### Observation Framing（観察者フレーミング）
「やれ」と言わず「こういう人がいる」と観察者として語る。
読者が「自分もそうしたい」と自発的に結論に達する。

✅ 「速くリリースできる開発者を見ていると、共通してアイデアが熱いうちに動ける仕組みを持っている」

### 受動的変化表現
命令形ではなく、**変化が自然に起きる**という表現を使う。

❌ 「スマホで開発しましょう」
✅ 「スマホで動ける回数が増えると、自然とリリースのペースが変わってくる」
✅ 「気づいたら、PCを開く前にもう手が動いていた」

### タイトル設計のPain × Benefit 公式

```
[Identity Signal（対象者を呼ぶ）] × [Pain（共感）] × [Benefit/Outcome（得たい未来）]
```

例：
- ❌ 「電車の中でアイデアが湧く。でもPCは家にある」（Pain のみ）
- ✅ 「アイデアが湧いた瞬間に形にできる人が、個人開発でひとり先を走っている」（Identity + Benefit + Curiosity Gap）

### セミナー語り口調（没入型コンテンツ）
読者を「文章を読む人」ではなく「話を聞いている人」にする。

- 問いかけ：「〜って、ありませんか？」「〜、想像してみてください」
- 共感の先回り：「自分も長い間これで悩んでいました」
- 驚きの予告：「でも、実際に触ってみたら全然違った」
- 気づきの演出：「正確に言うと——〜んですよね」

## AI適用ルール

- **命令形を避ける**：「〜しましょう」「〜してみよう」「ぜひ〜を」は使わない
- **Pain のみのタイトル禁止**：必ずベネフィットか Curiosity Gap を併置する
- **Future Pacing を必ず1箇所入れる**：読者が「その状態の自分」を映像として想像できるシーンを描く
- **結論は読者に「発見」させる**：観察者フレーミングで提示し、AIが言い切らない
- **受動的変化表現**：「自然と〜になっている」「気づいたら〜していた」で締める
