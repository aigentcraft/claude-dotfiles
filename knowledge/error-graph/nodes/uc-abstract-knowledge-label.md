## 1. Plan / What was asked
ユーザーはPDCAを回しながら一緒に成長するナレッジシステムの設計を求めていた。
具体的には：
- 作業中に発生したエラーの記録
- ユーザーからの指摘・追加リクエストが発生した理由の記録
- これらを蓄積して次回以降の行動改善に繋げる仕組み

## 2. Do / What Claude actually produced
Claudeはストレージ設計（daily/ と knowledge/ の2層構造）の説明として：
> 「重要な知見は knowledge/ に昇格（手動 or Claude判断）」

という抽象的な説明を返した。「重要な知見」とは何かを定義しなかった。

## 3. Check / Why the user had to correct

**correction_category: too-abstract**

「重要な知見」は概念として理解可能だが、**何を書けばいいか分からない**。
- 「エラーログを書く」→ 具体的、実行可能
- 「重要な知見を書く」→ 判断基準が曖昧、人によって解釈が違う

ユーザーの本当の意図は「AIが自分の失敗・ユーザーへの迷惑を構造的に記録し改善する仕組み」だったが、Claudeはそれをストレージ設計の問題として捉え、格納場所の話だけをした。

**本質のすれ違い**: 「何を保存するか」ではなく「どんな失敗をどう記録してどう改善するか」の設計が求められていた。

## 4. Act / Behavioral Rule for Next Time

**ルール1**: ナレッジシステム・記録システムを設計するとき、「何を書くか」を必ず具体例とカテゴリで示すこと。「重要な情報」「有用な知見」のような抽象ラベルは使用禁止。

**ルール2**: ユーザーが「一緒に成長する」「PDCAを回す」と言ったら、それは**AIの行動改善**の話である。ストレージ構造の話に終始してはいけない。

**ルール3**: システム設計の提案時は「誰が、いつ、何をトリガーに、何を書くか」まで具体化してから提案する。

**チェックリスト（次回のナレッジ設計時）**:
- [ ] 記録する内容を具体的なカテゴリで示したか？
- [ ] トリガー（いつ記録するか）を明示したか？
- [ ] AIの行動改善に繋がる仕組みになっているか？
